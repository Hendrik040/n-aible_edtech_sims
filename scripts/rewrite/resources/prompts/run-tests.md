## Testing skills to run

All of these run against the **experimental** environment —
never against `staging-v2` or `production-v3`.

### 1. API contract test — frontend-facing

Use the **api-contract-test** skill. It fetches the live experimental
OpenAPI spec, diffs it against your local backend, and lists any
endpoint whose shape changed (new / removed / shape-changed). It also
maps your git diff to router prefixes and calls out which endpoints
were plausibly affected by your changes.

Invoke it by asking to verify the API contract against the
experimental backend for this PR.

### 2. End-to-end simulation flow

Use the **e2e-simulation-flow** skill. It runs two linked flows
against the experimental environment:

- Flow 1 — Professor authors and publishes a simulation, creates a
  cohort, issues an invite (each mutating call cross-checked against
  Neon `experimental-v2` via the **neon-postgres** skill).
- Flow 2 — Fresh student registers, accepts the invite, plays
  through every scene with scripted Q&A from
  `fixtures/student-questions.json`.

Invoke it by asking for the full end-to-end simulation flow on
the experimental environment.

### 3. Daytona sandbox (conditional)

Only if your diff touches sandbox-related code
(`backend/common/services/sandbox_service.py`, the `execute-code`
endpoint, `scene_type=code_challenge` grading, or the
`UserProgress.sandbox_id` field). Use the **daytona-sandbox** skill
to probe current sandbox state and exercise a throwaway sandbox for
the affected code path. Clean up any sandbox you create in a
`trap`/`finally`.

## Workflow

1. Run the **api-contract-test** skill. Summarize what changed.
2. Run the **e2e-simulation-flow** skill.
3. If your diff touches sandbox paths, run the **daytona-sandbox**
   skill.
4. If any test is red:
   - For a contract regression: either fix the backend or update the
     contract fixture — whichever matches reality. Push a fix commit
     with message `{{TICKET_ID}}: fix <contract|E2E|sandbox> regression`.
   - For an E2E failure: read the response body in `run-e2e.py`'s
     output. Diagnose as either (a) broken endpoint — fix backend,
     (b) stale fixture — update `fixtures/student-questions.json`, or
     (c) flake — rerun once before concluding.
   - After pushing fixes, re-run the failing skill.

## Output the verdict (the loop reads this)

At the very end of your output, print exactly **one** of:

- `ALL_TESTS_PASSED` — contract + E2E (+ sandbox if applicable) are
  all green. The loop will proceed to merge the PR.
- `TESTS_FAILED: <short reason>` — you couldn't make them pass in
  this round. The loop will leave the PR open for manual review.

---

## Hard rules

- Never run tests against `staging-v2` or `production-v3` — only
  `experimental`.
- Never commit generated test-user credentials, session tokens, or
  anything from `~/.railway`, `~/.daytona`, `~/.config/neonctl`.
- Test users generated by the E2E flow are intentionally not cleaned
  up — leave them in the experimental DB for debugging.
- If a skill's pre-flight fails (e.g., neonctl not authed), stop and
  print `TESTS_FAILED: prerequisite <which one>` — don't fake a pass.
